# AI-агрегатор заявок

MVP сервиса из ТЗ: клиент описывает потребность на естественном языке, AI
классифицирует и уточняет параметры, система рассылает заявку подходящим
поставщикам вместе с контактом клиента — поставщики сами звонят и
договариваются напрямую, без сбора и сравнения предложений в системе.

Реализован весь обязательный (MVP) флоу из ТЗ и обсуждённые в переписке
UX-решения: чипы вместо ввода текста для перечислимых полей, геолокация/карта
для адресов (см. ограничения ниже), доверенное устройство для пропуска SMS
при повторных заявках, WebOTP для автоввода кода подтверждения.

## Стек

- **Backend**: NestJS + Prisma + PostgreSQL + Redis/BullMQ (волновая рассылка) + Socket.io
- **Frontend**: Next.js (App Router) + TypeScript + Tailwind
- **Shared**: `@ai-zayavki/shared` — статусы/переходы, типы категорий, события сокетов — общие для api и web, чтобы не разъезжались

## Структура

```
apps/
  api/     — NestJS backend
  web/     — Next.js frontend (клиент, поставщик, админка — всё в одном приложении)
packages/
  shared/  — общие типы и константы
```

## Быстрый старт

```bash
npm install
cp .env.example .env          # затем скопировать тот же файл в apps/api/.env — см. ниже
cp .env apps/api/.env
docker compose up -d          # Postgres :5433, Redis :6380, MinIO :9010/9011
npm run build -w packages/shared
npm run prisma:migrate        # применяет схему к БД
npm run prisma:seed           # 6 категорий MVP, тестовый админ, 4 демо-поставщика в Астане
npm run dev:api                # http://localhost:3001
npm run dev:web                # http://localhost:3000 (или PORT=3100 npm run start -w apps/web после build)
```

Почему `.env` нужен в двух местах: `apps/api/main.ts` явно грузит `.env` из
корня монорепо (так удобнее держать один файл на весь проект), а Prisma CLI
по умолчанию ищет `.env` рядом с `schema.prisma` / в cwd команды — поэтому
копия в `apps/api/.env` тоже нужна для `prisma migrate`/`prisma studio`.

Порты Postgres/Redis/MinIO в `docker-compose.yml` намеренно не дефолтные
(5433/6380/9010/9011) — подберите свои, если конфликтуют с чем-то ещё на
машине.

Тестовый админ после сида: `admin@example.com` / `admin12345`.

## Что реально работает (проверено вручную, включая через браузер)

Полный путь клиента: лендинг → AI-диалог (mock-провайдер классифицирует и
извлекает поля из свободного текста прямо как в примере из ТЗ — "Завтра нужен
манипулятор... из Астаны в Косшы") → чипы для уточнения → карточка заявки →
подтверждение телефона (с пропуском SMS на доверенном устройстве) →
публикация → рассылка подходящим поставщикам с полным описанием и контактом
клиента → live-обновление страницы клиента без перезагрузки (Socket.io), пока
поставщики звонят напрямую → клиент сам закрывает заявку («Услугу оказали» /
«Отменить») → повтор заявки.

Тот же диалог полностью проходится и через WhatsApp (см. раздел ниже) —
webhook прогонялся вручную с payload'ами в формате GREEN-API: классификация,
уточняющие вопросы кнопками/пронумерованным списком, карточка на подтверждение,
публикация без SMS-кода, попадание в ту же рассылку — всё без отдельного кода,
поверх тех же `OrdersService`-методов, что и веб.

Админка: логин, очереди заявок (активные/требуют внимания/отменённые),
повторная рассылка, отмена, список/блокировка поставщиков, настройки
рассылки.

## Приём заявок через WhatsApp (GREEN-API)

Второй вход в тот же `OrdersService` — работает поверх [green-api.com](https://green-api.com).
Номер телефона уже подтверждён самим WhatsApp, поэтому SMS-код при публикации
не запрашивается (см. `AuthOtpService.getOrCreateClientAuthUser`).

**Настройка:**
1. Завести инстанс на console.green-api.com — получите `idInstance` и `apiTokenInstance`.
2. Поднять API по публичному HTTPS (для локальной разработки — туннель, например `ngrok http 3001`).
3. Один раз вызвать `SetSettings`, указав ваш вебхук и токен (тот же, что в `WHATSAPP_WEBHOOK_TOKEN`):
   ```bash
   curl -X POST "https://api.green-api.com/waInstance{idInstance}/setSettings/{apiTokenInstance}" \
     -H "Content-Type: application/json" \
     -d '{"webhookUrl":"https://<ваш-домен>/whatsapp/webhook","webhookUrlToken":"<WHATSAPP_WEBHOOK_TOKEN>","incomingWebhook":"yes"}'
   ```
4. В `.env`: `WHATSAPP_PROVIDER=green-api`, заполнить `GREEN_API_ID_INSTANCE`/`GREEN_API_TOKEN_INSTANCE`.

Без этой настройки (`WHATSAPP_PROVIDER=console`, значение по умолчанию) всё
работает локально — исходящие сообщения просто пишутся в лог API, входящие
можно сэмулировать вручную POST'ом на `/whatsapp/webhook` с телом в формате
GREEN-API (`typeWebhook: "incomingMessageReceived"`) и заголовком
`Authorization: Bearer <WHATSAPP_WEBHOOK_TOKEN>`.

Правила рендеринга чипов в WhatsApp (`whatsapp/whatsapp-message-render.util.ts`):
не больше 3 кнопок в сообщении (ограничение платформы) — если вариантов
больше, уходит пронумерованный текстовый список, и ответ и кнопкой, и голым
числом обрабатываются одинаково.

## Регистрация поставщика через WhatsApp

Тот же бот, второй сценарий (`whatsapp/whatsapp-onboarding.service.ts`).
Точный триггер — сообщение целиком должно быть одной из фраз: «поставщик»,
«регистрация», «стать исполнителем», «мои услуги», «я поставщик» (намеренно
не подстрокой — иначе заказ вроде "Ищу поставщика песка" уводил бы клиента в
регистрацию поставщика). В проде вместо угадывания по тексту разумнее
отдельная wa.me-ссылка с заранее подставленным текстом.

Шаги: название компании (текст) → категории услуг (пронумерованный список,
переключение по одной с ✅, «Готово» для завершения) → города (текст через
запятую) → срочные заказы (Да/Нет) → карточка на подтверждение. Номер уже
верифицирован WhatsApp, поэтому регистрация завершается сразу — новый
профиль помечается `needsReview: true` и виден в админке (вкладка
«Поставщики», значок «новый, не проверен», кнопка «Проверено»). Тот же
триггер на уже зарегистрированный номер открывает тот же диалог заново с
подставленными текущими значениями — это одновременно и повторная
регистрация, и редактирование профиля.

## Рассылка заявок поставщикам (лид-модель)

Никакого сбора и сравнения предложений внутри системы — заявка просто
рассылается подходящим поставщикам вместе с контактом клиента, дальше они
сами звонят и договариваются напрямую. Раньше в MVP была отдельная модель с
предложениями/выбором клиента (`Offer`) и переключаемый режим авто-клейма —
от обоих отказались в пользу этой более простой схемы.

- `MatchingService.sendWave` при публикации (и при ручном «Повторить
  рассылку» из админки) находит до `DispatchSettings.waveSize` подходящих по
  категории/городу поставщиков, ещё не получавших эту заявку
  (`DispatchWave.supplierIds` копится между рассылками), и каждому шлёт
  **полное** описание заявки + телефон клиента одним сообщением
  (`order_broadcast_full` в `notification-templates.ts`). В WhatsApp номер
  телефона превращается в кликабельную ссылку автоматически, отдельных
  кнопок «Беру заказ» больше нет.
- Тот же контакт клиента виден и на веб-карточке заявки для поставщика
  (`/s/[orderId]`) — без входа/подтверждения телефона, доступ достаточно
  знать неугадываемую ссылку, ровно как раньше было устроено сравнение
  предложений.
- Если подходящих поставщиков не нашлось совсем — заявка уходит в статус
  `NEEDS_OPERATOR`, оператор разбирается вручную и может запустить рассылку
  ещё раз («Повторить рассылку» в админке), это же действие подхватывает и
  заявки, где часть поставщиков уже была уведомлена.
- Клиент сам закрывает заявку, когда услугу оказали (или нет) — кнопка
  «Услугу оказали» на странице заявки, один запрос `POST
  /orders/:id/complete`, без отдельного шага подтверждения от поставщика
  (его в системе больше никто не отслеживает — контакт открыт, дальше всё
  происходит вне системы).
- Статусная модель заявки заметно проще: `DRAFT → CLARIFYING →
  AWAITING_PHONE_CONFIRMATION → PUBLISHED → COMPLETED /
  CANCELLED_BY_CLIENT / CANCELLED_BY_ADMIN`, плюс `NEEDS_OPERATOR` как
  развилка из `PUBLISHED` (см. `packages/shared/src/statuses.ts`).

### Проактивная проверка статуса — раз клиент сам не закрывает заявку

Раз никто в системе не знает, кого именно клиент нашёл, нельзя просто ждать,
что клиент сам вернётся и нажмёт «Услугу оказали» — особенно если общение
шло только в WhatsApp и клиент туда просто не вернулся. Поэтому `publish()`
сразу ставит два отложенных job'а в ту же очередь `matching`
(`OrdersService.sendCompletionCheckin` / `escalateStaleOrder`,
обрабатываются в `MatchingProcessor`):

- через `ORDER_CHECKIN_DELAY_HOURS` (по умолчанию 24ч) — если заявка всё ещё
  `PUBLISHED`, клиенту уходит сообщение «Удалось решить вопрос?» с кнопками
  Да/Нет (`complete|yes|<id>` / `complete|no|<id>`) в WhatsApp, на SMS/console
  — тот же текст со ссылкой на страницу заявки (кнопок платформа не
  поддерживает);
- через `ORDER_CHECKIN_ESCALATE_HOURS` (по умолчанию 48ч) — если заявка всё
  ещё `PUBLISHED` (клиент не ответил вообще) — она уходит в
  `NEEDS_OPERATOR` автоматически, оператор видит её в очереди «Требуют
  внимания» и связывается сам.

Нажатие кнопки в WhatsApp обрабатывается на верхнем уровне роутера
(`whatsapp-router.service.ts`, `complete|` префикс) независимо от того, в
каком состоянии диалог — так же, как раньше был устроен приём оплаты
подписки. Кроме того, если клиент напишет боту что угодно, пока его заявка
активна (`PUBLISHED`), в ответ вместо простого статуса приходят те же две
кнопки — это закрывает и второй сценарий: клиент сам спрашивает «закрыта ли
заявка».

Проверено вручную (с временно укороченными `ORDER_CHECKIN_*_HOURS` для
скорости): чек-ин приходит по таймеру, кнопка «Да» переводит в `COMPLETED`,
кнопка «Нет» — в `NEEDS_OPERATOR` с уведомлением, повторное нажатие по уже
закрытой заявке отвечает вежливой ошибкой вместо падения, отсутствие ответа
приводит к автоэскалации по второму таймеру.

## Платная подписка на уведомления

Первые `FREE_NOTIFICATIONS_PER_MONTH` (по умолчанию 10) заявок в календарный
месяц поставщик получает бесплатно; дальше `MatchingService.sendWave`
пропускает его при рассылке, пока не оформлена подписка — вместо самой
заявки (не чаще раза в сутки) уходит напоминание со ссылкой на оплату.
Оплачивает поставщик, разговор об оплате — в WhatsApp: команда «баланс»
показывает остаток бесплатных заявок и статус подписки, кнопка «Оформить
подписку» присылает ссылку.

Kaspi Pay пока не подключён (нужен реальный договор с Kaspi — это
организационный шаг, не инженерный) — `PAYMENT_PROVIDER=mock` (по умолчанию)
эмулирует оплату: ссылка на оплату — это `GET /billing/mock-confirm/:reference`
на самом же API, открыть её и есть «заплатить». Переключение на прод —
`PAYMENT_PROVIDER=kaspi` плюс реализация `billing/kaspi-payment.provider.ts`
(сейчас там заглушка по тому же паттерну, что `sms/mobizon-sms.provider.ts`).

Счётчик сбрасывается 1-го числа каждого месяца cron-задачей
(`BillingService.resetMonthlyQuotas`, `@nestjs/schedule`). Оператор может
вручную выдать/снять подписку в админке («Поставщики» → «Выдать
подписку») — например, для форс-мажора или тестов, без реальной оплаты.

Проверено вручную: 11 волн подряд на одного поставщика — 10 прошли, 11-я
заблокирована с напоминанием (и кнопкой в WhatsApp), мок-оплата по ссылке
сразу активирует подписку на `SUBSCRIPTION_PERIOD_DAYS` (30) дней, после
чего рассылка снова доходит без учёта лимита; ручная выдача/снятие в
админке отражается в списке поставщиков сразу.

## Подключаемые внешние интеграции (dev-заглушки по умолчанию)

Каждая внешняя зависимость спрятана за интерфейсом и переключается одной
переменной окружения — можно гонять всё локально без единого реального ключа,
и лишь потом подставить настоящего провайдера.

| Что | Переменная | Dev-режим | Продакшен |
|---|---|---|---|
| SMS | `SMS_PROVIDER` | `console` — код пишется в лог API | `mobizon` (заготовка в `sms/mobizon-sms.provider.ts`, нужен реальный API-ключ и доработка запроса) |
| AI | `AI_PROVIDER` | `mock` — детерминированный классификатор на ключевых словах и regex-экстрактор | `openai` — структурированный вывод через `AI_PROVIDER=openai` + `OPENAI_API_KEY` |
| Фото | `STORAGE_PROVIDER` | `local` — пишет в `apps/api/uploads`, отдаёт статикой | `s3` — реальный S3-совместимый (работает и с MinIO из docker-compose) |
| Карты/геокодинг | `MAPS_PROVIDER` | `none` — адрес хранится как обычный текст | не реализовано, только точка расширения |
| WhatsApp | `WHATSAPP_PROVIDER` | `console` — исходящие пишутся в лог API | `green-api` — см. раздел про WhatsApp выше |
| Платежи (подписка) | `PAYMENT_PROVIDER` | `mock` — оплата эмулируется одним GET-запросом | `kaspi` (заготовка в `billing/kaspi-payment.provider.ts`, нужен договор с Kaspi + доработка запроса) |

## Осознанные упрощения MVP (не баги, но стоит знать)

- **Гео пока текстовое, не координатное.** Сопоставление поставщика с
  заявкой идёт по точному совпадению строки `city`, без district/радиуса и
  без реального геокодинга. Для demo-провайдера AI это нормализовано под
  несколько городов КЗ (`ai/mock-ai.provider.ts`), для реального OpenAI —
  зависит от модели.
- **Точный адрес виден сразу, без раскрытия по этапам.** П.10.2 ТЗ про
  "показывать район, а не точный адрес до выбора" был актуален для модели со
  сравнением предложений; в лид-модели контакт и полное описание заявки и
  так уходят поставщику одним сообщением, разделять по уровням детализации
  больше незачем.
- **Кто именно выполнил заказ, система не знает.** Раз поставщики
  связываются с клиентом напрямую, в системе не осталось привязки
  завершённой заявки к конкретному поставщику — `SupplierProfile.rating`/
  `completedOrders` больше не пересчитываются автоматически.
- **Комиссия/оплата/безопасная сделка** — сознательно не в скоупе, в самом
  ТЗ это "следующий этап".

## Дальнейшие шаги, если продолжать

1. Реальный SMS-провайдер под Казахстан (Mobizon и т.п.) — единственный
   доработанный адаптер, интерфейс уже готов.
2. Геокодинг адресов (2GIS API) — сейчас `MAPS_PROVIDER=none`.
3. Все 4 задачи из плана (приём заявки через WhatsApp, рассылка с прямым
   контактом клиента, регистрация поставщика через WhatsApp, платная
   подписка на уведомления) реализованы — см. разделы выше. Из них не
   хватает только реального договора с Kaspi (Pay) — организационный шаг,
   не инженерный; на моке всё работает end-to-end.
4. Полноценный редактор шаблонов категорий в админке (сейчас поля
   редактируются через `PATCH /admin/categories/:id` с сырым JSON).
5. Продовый деплой: собрать `apps/api` и `apps/web` (`npm run build`),
   поднять за реверс-прокси, вынести секреты из `.env` в secret manager.
